# CD: GitHub Actions로 OCI Ampere 배포 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** `master`에 push하면 GitHub Actions가 테스트를 돌리고, 통과하면 소스를
OCI Ampere 인스턴스(`146.56.170.98`)로 rsync한 뒤 서버에서 빌드·마이그레이션·
systemd 재시작까지 자동으로 수행하게 만든다.

**Architecture:** CI 러너(x86_64)에서는 checkout과 rsync만 하고, 실제
`npm ci`/`prisma migrate deploy`/`npm run build`는 서버(ARM, A1.Flex)에서
실행한다. 서버 프로세스는 sudo 없이 systemd **user** 서비스로 관리한다.

**Tech Stack:** GitHub Actions, rsync over SSH, bash, systemd (user unit),
Next.js 16 / Prisma 7 (기존 스택 그대로).

## Global Constraints

- 커밋 메시지는 한국어로, conventional-commit 타입 접두사만 영어로 유지
  (`feat:`, `fix:`, `docs:` 등). 태스크당 커밋 1개.
- `.env` 및 시크릿을 담은 파일은 절대 git에 커밋하지 않는다.
- rsync는 서버의 `.env`를 덮어쓰거나 삭제하지 않는다 (제외 + `--delete`가
  제외 항목엔 적용되지 않는 rsync 기본 동작에 의존).
- systemd 유닛은 **user** 단위로 설치한다 (`~/.config/systemd/user/`),
  sudo 불필요.
- `master`가 기본 작업 브랜치이며 별도 feature 브랜치를 만들지 않는다
  (레포 규칙).

### Task 2: 서버용 배포 스크립트 작성

**Files:**
- Create: `scripts/deploy-remote.sh`

**Interfaces:**
- Consumes: `deploy/drillup.service` (Task 1의 템플릿, `__DEPLOY_PATH__`
  토큰 포함)
- Consumes env var: `DEPLOY_PATH` (필수, 서버 상 앱 디렉터리 절대경로)
- Produces: `~/.config/systemd/user/drillup.service` (최초 실행 시 생성),
  `drillup` systemd user 서비스가 `active (running)` 상태가 됨

- [ ] **Step 1: 스크립트 작성**

```bash
#!/usr/bin/env bash
set -euo pipefail

: "${DEPLOY_PATH:?DEPLOY_PATH env var must be set}"

cd "$DEPLOY_PATH"

npm ci
npx prisma migrate deploy
npm run build

UNIT_DIR="$HOME/.config/systemd/user"
UNIT_PATH="$UNIT_DIR/drillup.service"

mkdir -p "$UNIT_DIR"

if [ ! -f "$UNIT_PATH" ]; then
  sed "s|__DEPLOY_PATH__|$DEPLOY_PATH|g" "$DEPLOY_PATH/deploy/drillup.service" > "$UNIT_PATH"
  systemctl --user daemon-reload
  systemctl --user enable drillup
fi

systemctl --user restart drillup
systemctl --user status drillup --no-pager
```

Save this content to `scripts/deploy-remote.sh`, then make it executable:

Run: `chmod +x scripts/deploy-remote.sh`

- [ ] **Step 2: 문법 검사**

Run: `bash -n scripts/deploy-remote.sh`
Expected: 출력 없이 종료 (문법 오류 없음)

- [ ] **Step 3: 가짜 서버 환경으로 드라이런 테스트 작성**

실제 Ampere 서버 없이도 스크립트 로직(템플릿 치환, 최초 설치 시에만
유닛 생성하는 멱등성)을 검증하기 위해 `npm`/`npx`/`systemctl`을 흉내 내는
가짜 바이너리를 깔고 스크립트를 두 번 실행한다.

아래 내용을 `/tmp/drillup-cd-test/run.sh`에 저장:

```bash
#!/usr/bin/env bash
set -euo pipefail

ROOT=/tmp/drillup-cd-test
rm -rf "$ROOT"
mkdir -p "$ROOT/bin" "$ROOT/home" "$ROOT/app/scripts" "$ROOT/app/deploy"

cp "$OLDPWD/scripts/deploy-remote.sh" "$ROOT/app/scripts/deploy-remote.sh"
cp "$OLDPWD/deploy/drillup.service" "$ROOT/app/deploy/drillup.service"

for cmd in npm npx systemctl; do
  cat > "$ROOT/bin/$cmd" <<EOF
#!/usr/bin/env bash
echo "$cmd \$*"
EOF
  chmod +x "$ROOT/bin/$cmd"
done

export HOME="$ROOT/home"
export PATH="$ROOT/bin:$PATH"
export DEPLOY_PATH="$ROOT/app"

echo "--- 1st run ---"
bash "$ROOT/app/scripts/deploy-remote.sh"

echo "--- unit file after 1st run ---"
cat "$ROOT/home/.config/systemd/user/drillup.service"

echo "--- 2nd run (idempotency check) ---"
bash "$ROOT/app/scripts/deploy-remote.sh"
```

Run: `bash /tmp/drillup-cd-test/run.sh`

Expected:
- `--- 1st run ---` 아래에 `npm ci`, `npx prisma migrate deploy`,
  `npm run build`, `systemctl daemon-reload`, `systemctl enable drillup`,
  `systemctl restart drillup`, `systemctl status drillup --no-pager`가
  순서대로 출력됨
- 유닛 파일 내용에 `WorkingDirectory=/tmp/drillup-cd-test/app`가 보임
  (`__DEPLOY_PATH__` 토큰이 실제 경로로 치환됨)
- `--- 2nd run ---` 아래에는 `systemctl enable drillup`이 **다시 나타나지
  않고** `systemctl restart drillup`만 나타남 (이미 유닛 파일이 있으므로
  재설치를 건너뜀)

이 드라이런은 임시 검증용이며 레포에 커밋하지 않는다.

### Task 3: GitHub Actions 워크플로우 작성

**Files:**
- Create: `.github/workflows/deploy.yml`

**Interfaces:**
- Consumes: repo secrets `SSH_HOST`, `SSH_USER`, `SSH_PRIVATE_KEY`,
  `DEPLOY_PATH` (필수), `SSH_PORT` (선택, 없으면 22)
- Consumes: Task 2의 `scripts/deploy-remote.sh` (서버에 rsync된 뒤 SSH로
  실행됨)

- [ ] **Step 1: 워크플로우 작성**

```yaml
name: Deploy

on:
  push:
    branches: [master]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
      - run: npm ci
      - run: npm run lint
      - run: npm test

  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Configure SSH
        env:
          SSH_PRIVATE_KEY: ${{ secrets.SSH_PRIVATE_KEY }}
          SSH_HOST: ${{ secrets.SSH_HOST }}
          SSH_PORT: ${{ secrets.SSH_PORT }}
        run: |
          mkdir -p ~/.ssh
          printf '%s\n' "$SSH_PRIVATE_KEY" > ~/.ssh/deploy_key
          chmod 600 ~/.ssh/deploy_key
          ssh-keyscan -p "${SSH_PORT:-22}" "$SSH_HOST" >> ~/.ssh/known_hosts

      - name: Rsync source to server
        env:
          SSH_HOST: ${{ secrets.SSH_HOST }}
          SSH_USER: ${{ secrets.SSH_USER }}
          SSH_PORT: ${{ secrets.SSH_PORT }}
          DEPLOY_PATH: ${{ secrets.DEPLOY_PATH }}
        run: |
          rsync -avz --delete \
            -e "ssh -i ~/.ssh/deploy_key -p ${SSH_PORT:-22}" \
            --exclude='.git/' \
            --exclude='node_modules/' \
            --exclude='.next/' \
            --exclude='.env' \
            --exclude='generation_output/' \
            --exclude='generation_reference/' \
            --exclude='tsconfig.tsbuildinfo' \
            ./ "$SSH_USER@$SSH_HOST:$DEPLOY_PATH/"

      - name: Run remote deploy script
        env:
          SSH_HOST: ${{ secrets.SSH_HOST }}
          SSH_USER: ${{ secrets.SSH_USER }}
          SSH_PORT: ${{ secrets.SSH_PORT }}
          DEPLOY_PATH: ${{ secrets.DEPLOY_PATH }}
        run: |
          ssh -i ~/.ssh/deploy_key -p "${SSH_PORT:-22}" "$SSH_USER@$SSH_HOST" \
            "DEPLOY_PATH='$DEPLOY_PATH' bash '$DEPLOY_PATH/scripts/deploy-remote.sh'"
```

Save this content to `.github/workflows/deploy.yml`.

### 필요한 GitHub Secrets

레포 Settings -> Secrets and variables -> Actions에 등록:

- `SSH_HOST` - 배포 대상 서버 IP
- `SSH_USER` - SSH 접속 계정
- `SSH_PRIVATE_KEY` - 그 계정의 SSH 개인키 전체 내용
- `DEPLOY_PATH` - 서버 상 앱 디렉터리 절대경로 (홈 디렉터리 하위, 예:
  `/home/<user>/drillup`)
- `SSH_PORT` - SSH 포트 (선택, 없으면 22)

### 서버 측 1회성 사전 준비 (최초 배포 전에 수동으로)

- Node 22+ 설치
- `DEPLOY_PATH`에 프로덕션 `.env` 파일 직접 생성 (`DB_*`,
  `APP_PASSWORD`, `SESSION_SECRET`, `NODE_ENV=production`, `PORT=3000`)
- `loginctl enable-linger <user>` 실행 (재부팅 후에도 systemd user
  서비스가 유지되도록)
- 이 서버의 MariaDB 접속 정보를 `.env`의 `DB_*` 값에 반영

HTTPS 리버스 프록시(nginx 등) 구성은 이 워크플로우 범위 밖이며 별도로
진행한다.
