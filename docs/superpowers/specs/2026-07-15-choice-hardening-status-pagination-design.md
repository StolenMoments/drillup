# 선지 검토 상태별 페이지네이션 설계

날짜: 2026-07-15

## 목적

`/hardening`이 모든 선지 강화 작업을 한 번에 내려받아 렌더링하는 구조를 상태별
요약과 서버 페이지네이션으로 바꾼다. 요약 화면은 빠르게 현황을 파악하는 용도로
유지하고, 전체 이력은 독립 상세 경로에서 확인한다.

## 확정된 동작

- 요약은 승인 대기, 진행 중, 실패, 반영 이력의 최신 5건과 전체 건수를 반환한다.
- 상세는 상태별 10건을 서버에서 `count` 후 `skip/take`로 조회한다.
- 상태 상세 경로는 `/hardening/pending`, `/running`, `/failed`, `/applied`다.
- 유효하지 않은 상태 경로는 404, API 상태 값은 400으로 처리한다.
- 페이지가 없거나 잘못되면 1, 마지막 페이지를 넘으면 마지막 페이지로 보정한다.
- 거절된 작업은 기존처럼 대기·실패 목록에서 제외하고, 반영 이력은 전체를 조회한다.

## API 계약

- `GET /api/harden-jobs`
  - 각 상태의 `{ items, totalItems }`를 담은 요약을 반환한다.
- `GET /api/harden-jobs?status=<status>&page=<page>`
  - `{ items, page, pageSize: 10, totalItems, totalPages }`를 반환한다.

상태별 Prisma 조건과 정렬은 서비스의 공통 정의 하나를 요약과 상세가 공유한다.
요약은 상태마다 `take: 5`, 상세는 `count` 결과로 페이지를 보정한 뒤 `skip/take`를
적용한다.

## 화면과 동시성

요약의 각 섹션은 전체 건수와, 5건을 초과할 때 `전체 N건 보기` 링크를 표시한다.
상세는 기존 문제 목록과 같은 이전/현재·전체 페이지/표시 범위/다음 컨트롤을 쓴다.
승인·거절·재생성 후 현재 페이지를 다시 조회하며, 마지막 항목 제거로 서버가 페이지를
보정하면 `router.replace`로 URL도 교정한다.

요약과 상세은 보이는 탭에서만 5초마다 갱신한다. 탭 복귀 시 즉시 갱신하며, 요청마다
순번을 부여해 늦게 도착한 과거 응답이 최신 화면을 덮지 못하게 한다.

## 조회 인덱스

기존 stale-job 인덱스는 유지하고 다음 인덱스를 추가한다.

- 대기: `(status, applied_at, dismissed_at, finished_at)`
- 실패: `(status, dismissed_at, finished_at)`
- 진행: `(status, created_at)`
- 반영: `(applied_at)`

## 테스트

서비스 필터·정렬·건수·페이지 보정, API 분기·검증, 요약/상세 렌더링, 액션 후
재조회와 URL 보정, 탭 가시성 및 오래된 응답 무시를 Vitest로 검증한다.
