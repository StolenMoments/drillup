# 로그인 제한 DoS 방지 설계

## 목표

현재 모듈 전역 로그인 실패 카운터를 제거해 외부인이 모든 사용자의 로그인을 잠그는 서비스 거부 취약점을 없앤다. 동시에 앱과 nginx 두 계층에서 IP별 브루트포스 방어를 유지한다.

## 확인된 문제

`src/server/login-throttle.ts`는 모든 요청이 하나의 `failureCount`와 `lockedUntil`을 공유한다. `src/app/api/auth/login/route.ts`는 비밀번호 확인 전에 이 상태를 검사하므로, 임의의 외부인이 실패 요청 5회를 보내면 올바른 비밀번호도 15분간 거부된다. 상태가 프로세스 메모리에만 있어 재배포 시 초기화되고 여러 인스턴스 사이에서 공유되지 않는 문제도 있다.

## 범위

- 앱의 전역 로그인 락아웃을 IP별 제한으로 교체한다.
- 올바른 비밀번호는 해당 IP가 제한 중이어도 허용한다.
- 잘못된 비밀번호에만 IP별 지수 backoff를 적용한다.
- 오래 사용되지 않은 IP 상태를 정리해 메모리 증가를 제한한다.
- nginx 로그인 경로에 클라이언트 IP별 요청 제한을 추가한다.
- 저장소 설정과 운영 서버 설정을 함께 반영하고 검증한다.

영구 저장소를 추가하거나 여러 앱 인스턴스의 제한 상태를 공유하는 기능은 현재 단일 프로세스 운영 범위에 포함하지 않는다. nginx의 공유 메모리 zone이 여러 nginx worker 사이의 1차 제한을 담당한다.

## 앱 설계

### IP 식별

로그인 Route Handler는 nginx가 `$remote_addr`로 덮어쓰는 `X-Real-IP` 헤더를 클라이언트 키로 사용한다. 헤더가 없거나 유효한 단일 IP 문자열이 아니면 고정 키 `unknown`을 사용한다. 임의의 `X-Forwarded-For` 체인은 신뢰하지 않는다.

### 제한 상태

`src/server/login-throttle.ts`는 IP 키마다 다음 상태를 보관한다.

- 연속 실패 횟수
- 다음 잘못된 비밀번호 시도를 허용하는 시각
- 마지막 접근 시각

실패 시 대기시간은 `1초, 2초, 4초, 8초, 16초`로 증가하고 이후 30초로 제한한다. 제한 시간이 지난 뒤 다시 실패하면 다음 단계의 대기시간을 적용한다. 성공하면 해당 IP의 상태만 삭제한다.

마지막 접근 후 30분이 지난 항목은 요청 처리 중 정리한다. 정리 후에도 항목 수가 상한을 넘으면 가장 오래된 항목부터 제거한다. 초기 구현의 상한은 10,000개로 둔다.

### 로그인 요청 흐름

1. 요청 본문을 검증하고 비밀번호를 비교한다.
2. 비밀번호가 맞으면 해당 IP 상태를 초기화하고 세션을 발급한다.
3. 비밀번호가 틀리고 해당 IP가 backoff 중이면 `429`와 `Retry-After`를 반환한다.
4. 제한 중이 아니면 실패를 기록하고 기존 `401 INVALID_PASSWORD`를 반환한다.

비밀번호 비교를 제한 검사보다 먼저 수행하므로 공격자가 만든 실패 상태가 올바른 비밀번호 로그인을 차단하지 않는다. 사용자 대상 기존 한국어 문구와 이모지는 유지한다.

## nginx 설계

저장소의 `deploy/nginx/drillup.mygreed.shop.conf`에 `$binary_remote_addr` 기반 로그인 전용 zone을 정의한다. `/api/auth/login` exact-match location에 `limit_req`를 적용한 뒤 기존 프록시 헤더와 타임아웃을 동일하게 전달한다.

- 지속 속도: 분당 10회
- 순간 burst: 5회
- 초과 응답: `429`

일반 `location /`에는 제한을 적용하지 않아 학습·문제 생성 API에 영향을 주지 않는다. 운영 반영은 임시 파일에 복사한 뒤 `sudo nginx -t`가 성공한 경우에만 정식 설정으로 교체하고 reload한다.

## 오류 처리

- 앱 backoff 응답은 기존 오류 형식 `{ "error": { "code", "message" } }`을 유지한다.
- 앱의 `429`에는 남은 초를 올림한 `Retry-After`를 포함한다.
- nginx 제한 응답도 HTTP 상태는 `429`로 맞춘다. nginx 자체 응답 본문은 앱 JSON 형식의 적용 대상이 아니다.
- nginx 설정 검사가 실패하면 reload하지 않고 기존 운영 설정을 유지한다.

## 테스트 및 검증

TDD로 다음 회귀 테스트를 먼저 실패시키고 구현한다.

- IP A의 반복 실패가 IP B를 제한하지 않는다.
- 제한 중인 IP도 올바른 비밀번호로 로그인할 수 있다.
- 실패 간격이 지수형으로 증가하고 30초에서 제한된다.
- 성공하면 해당 IP 상태만 초기화된다.
- 만료 상태와 최대 항목 수가 정리된다.
- `Retry-After`가 남은 제한 시간을 반영한다.

로컬에서는 단일 테스트, 전체 `npm test`, `npm run lint`, `npm run build`를 실행한다. nginx 설정은 저장소 파일 검토와 운영 서버의 `sudo nginx -t`로 검증한다. 반영 후 `systemctl --user is-active drillup`, nginx 상태, 정상 로그인 경로의 비파괴적인 HTTP 응답을 확인한다. 실제 비밀번호 실패 요청을 반복해 운영 계정을 시험하지 않는다.

## 커밋 및 배포

설계 문서, 앱 변경, nginx 변경은 각각 작업 단위의 한국어 conventional commit으로 남긴다. 기존 사용자 작업 파일과 비밀 파일은 커밋하지 않는다. 검증된 커밋을 `master`에 push해 앱을 배포하고, nginx 설정은 SSH로 별도 반영한다.
