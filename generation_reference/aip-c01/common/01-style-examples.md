# AIP-C01 공식 샘플 문항 스타일 요약

> 스냅샷: 2026-07 기준  
> 출처 기반: `docs/AIP-C01 샘플 문제.txt`  
> 용도: 문항 원문 복제가 아니라 문제 생성 스타일, 오답 구성 방식, 정답 근거 패턴을 맞추기 위한 참고 자료

## 문항 톤

- Professional 레벨답게 긴 비즈니스 시나리오가 먼저 제시된다.
- 회사, 팀, 개발자, 금융 서비스, 교육, 의료, 고객 지원 같은 조직 맥락이 자주 등장한다.
- 요구 사항은 보통 3개 이상이다. 예: 운영 오버헤드 최소화, 지연 시간 제한, 보안/규정 준수, 비용 효율성, 기존 아키텍처 유지.
- 질문은 “가장 운영 효율적인 방식”, “최소한의 운영 오버헤드”, “가장 비용 효율적인 솔루션”, “요구 사항을 충족하는 단계 조합”처럼 최적 선택을 요구한다.
- 단일 선택과 복수 선택이 섞인다. 복수 선택 문항은 “2개 선택”처럼 선택 개수를 명시한다.

## 정답 패턴

- AWS 관리형 기능을 직접 사용하는 선택지가 정답인 경우가 많다.
- 사용자 지정 Lambda 프록시, 수동 승인, 자체 랭킹 알고리즘, 자체 서버 구성은 운영 오버헤드가 커서 오답이 되는 경우가 많다.
- 요구 사항의 핵심 제약을 정확히 만족하는 서비스 한계가 중요하다. 예: GPU 필요 여부, 페이로드 크기, 15분 처리 시간, 실시간/비동기/배치 추론의 차이.
- IAM 조건 키, S3 Object Lock, Bedrock 모델 호출 로깅, Bedrock Guardrails, Prompt Management, Knowledge Bases, 리랭커 모델처럼 특정 기능을 정확히 적용하는 선택지가 강하다.
- “이미 AWS 서비스가 기본 제공하는 기능”을 우회해서 직접 구현하는 선택지는 대체로 운영 오버헤드 또는 장애 지점 때문에 약하다.

## 오답 패턴

- 요구 사항 중 하나만 만족하고 나머지를 놓친다.
- 기능 목적이 다른 서비스를 그럴듯하게 끼워 넣는다. 예: CloudTrail은 API 메타데이터에는 적합하지만 실제 프롬프트/응답 로깅 요구에는 부족할 수 있다.
- 보안이나 검토를 강화하지만 개발 속도, 자동화, 생산성을 떨어뜨린다.
- 사용자 지정 코드, 수동 프로세스, 별도 인프라 운영이 필요해서 “최소 운영 오버헤드” 조건에 맞지 않는다.
- 실시간/비동기/배치, 서버리스/GPU 지원, 벡터 검색 지원 여부 같은 서비스 제한을 위반한다.
- 로그, 모니터링, X-Ray 성공 여부가 제시된 문제에서 원인 진단을 잘못한다. 검색 문제와 생성 문제를 구분해야 한다.

## 샘플에서 확인된 주제 앵커

- Amazon Q Developer: 코드 생성, 리팩터링, API 사용 지침, 성능 최적화, 테스트 생성으로 개발 생산성을 높인다. 모든 변경의 수동 승인이나 사후 문서화 중심 접근은 속도 요구와 맞지 않는다.
- Amazon Bedrock Guardrails: `bedrock:GuardrailIdentifier` 조건 키를 IAM 정책에 사용해 InvokeModel/Converse 호출에 가드레일 사용을 강제할 수 있다.
- Amazon Bedrock AgentCore Runtime: AgentCore SDK의 `@app.entrypoint`와 스타터 도구 키트는 서버 설정, 엔드포인트, 패키징, 컨테이너화, 배포 오버헤드를 줄인다.
- Bedrock 추론 파라미터: 특정 문구 이후 생성을 멈추려면 프롬프트 지시가 아니라 중지 시퀀스 파라미터를 사용한다.
- 데이터 계보: 생성 출력에 데이터 소스 메타데이터를 태그하고 입력 데이터를 AWS Glue Data Catalog에 등록하면 검토자가 소스 신뢰성을 확인하기 쉽다.
- 모델 호출 감사: 실제 프롬프트와 응답을 보존해야 하면 Bedrock 모델 호출 로깅을 S3 대상으로 설정하고 S3 Object Lock을 적용하는 패턴이 강하다.
- 벡터 검색 비용: 검색 빈도가 낮고 대규모 벡터를 저장해야 하는 경우 S3 Vectors 같은 서버리스/사용량 기반 접근이 비용 효율적일 수 있다.
- 검색 관련성: Bedrock 리랭커 모델과 하이브리드 검색은 RAG 결과 순위를 개선하는 관리형 접근이다.
- 임베딩 드리프트: 문서 인덱싱에 사용한 임베딩 모델과 쿼리 임베딩 모델 버전이 다르면 검색 품질이 급격히 나빠질 수 있다.
- SageMaker 추론 배포: GPU, 큰 입력, 최대 15분 응답 요구는 비동기식 추론 엔드포인트가 맞고, 서버리스 추론은 GPU 요구를 만족하지 못할 수 있다.

## 문제 작성 템플릿

### 단일 선택

1. 조직 맥락을 제시한다.
2. 현재 사용 중인 AWS 서비스와 제약을 제시한다.
3. 비용, 보안, 운영 오버헤드, 성능, 지연 시간, 데이터 크기 중 2~4개 조건을 명시한다.
4. “가장 적합한 솔루션은 무엇인가?”로 묻는다.
5. 보기 4개를 만든다. 정답은 관리형 기능과 요구 조건을 모두 만족한다. 오답은 그럴듯하지만 한 조건을 위반한다.

### 복수 선택

1. “다음 중 어떤 단계를 조합해야 하는가? (2개 선택.)” 형식으로 묻는다.
2. 정답 2개는 서로 보완적인 조치여야 한다.
3. 오답 3개는 운영 오버헤드 증가, 기능 목적 불일치, 보안/성능 요구 누락 중 하나를 포함한다.

## 해설 작성 기준

- 정답 해설은 왜 요구 사항을 만족하는지 서비스 기능과 연결한다.
- 오답 해설은 “왜 틀렸는지”를 한 문장으로 끝내지 말고, 어떤 요구 사항을 놓쳤는지 명확히 적는다.
- “최소 운영 오버헤드”가 키워드인 문항에서는 사용자 지정 구현, 수동 운영, 별도 인프라 관리가 왜 불리한지 설명한다.
- “비용 효율성”이 키워드인 문항에서는 사용 빈도, 프로비저닝된 용량, 사용량 기반 과금 차이를 설명한다.
